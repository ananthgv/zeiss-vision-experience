Production Roadmap

A practical guide for taking this prototype to production

Phase 1
Stakeholder Validation & Feedback
Before any backend work, validate the experience with the people who will use it every day.
PersonaWhy Their Input Matters
OptometristValidates clinical accuracy — Rx data presentation, VISUCORE/VISUFIT guidance, and whether the workflow matches real device operation
Dispensing OpticianThe primary daily user — tests frame selection, coating comparisons, pricing logic, and order placement UX
Store ManagerEvaluates operational fit — patient throughput, inventory accuracy, and order routing (HQ vs local POS)
Patient / ConsumerTests the "premium feel" — is it approachable? Does the AI recommendation build confidence or confusion?
Zeiss Sales / Account MgrCan this be used as a demo tool for pitching Zeiss equipment + lens bundles to new stores?
IT / Systems AdminEvaluates deployment feasibility — on-premise vs cloud, device integration, data security
Phase 2
Backend Integration Strategy
The app's stage-based architecture means each step can be wired to a real backend independently — no big-bang rewrite needed.
StageCurrent (Hardcoded)Production Backend
PatientLocal React statePatient Management System — REST API for create/lookup
RefractionSimulated scan + static RxVISUCORE 500 API — Query device DB for latest Rx
CentrationSimulated 3D scanVISUFIT 1000 API — Fetch avatar + centration data
LifestyleHardcoded coating listZeiss Product API — Dynamic coatings with pricing
LensJavaScript if/else logicAI/ML Service — POST Rx + lifestyle → ranked lenses
FramesStatic array of 6 framesFrame Catalog + Inventory API — Real-time stock
CheckoutNo-op button clickOMS + Payment Gateway — Fulfillment + transactions
Recommended
Cloud-First
SPA on CDN, APIs on Zeiss cloud, device data synced via secure gateway
+ Easy updates, centralized analytics
– Requires in-store internet
Alternative
On-Premise / Hybrid
Local server in-store, periodic cloud sync, LAN device integration
+ Works offline, data stays local
– Per-store maintenance
Best of Both
Progressive Web App
PWA with offline caching, syncs to cloud when connected
+ Resilient, installable on kiosks
– More complex service worker logic
Phase 3
Feature Enhancements
High Priority
  • Real device handshake — Poll VISUCORE / VISUFIT for scan completion instead of simulated timers
  • Dynamic product catalog — Lenses, coatings, frames from API with real pricing and images
  • Persistent patient sessions — Save progress so patients can pause and resume
  • Multi-language support — German, Mandarin, Japanese, Hindi for global stores
  • Print / PDF export — Professional prescription + order summary for the patient
Medium Priority
  • AR frame try-on — Virtual preview using VISUFIT avatar data
  • Staff authentication — Role-based access for optometrist vs optician
  • Analytics dashboard — Conversion rates, popular lenses, average order value
  • Appointment scheduling — Book follow-up fitting from order confirmation
  • Rx history — Returning patients see previous prescription for comparison
Nice to Have
  • Tablet-optimized layout — Most in-store kiosks are iPad-based
  • Dark mode — For stores with controlled lighting environments
  • Voice guidance — Accessibility for visually impaired patients
  • QR code sharing — Patient scans QR to view recommendation on their phone
Phase 4
Compliance & Security
HIPAA / GDPR
Patient health data encrypted at rest and in transit
Data Residency
Patient records stay in the correct geographic region
Audit Logging
Track who accessed or modified patient records
Consent Management
Patient opt-in for data storage and sharing
Phase 5
Go-to-Market
  • Pilot in 2–3 Zeiss Vision Centre locations
  • Measure — Patient satisfaction, staff efficiency, order accuracy, average transaction value
  • Iterate based on pilot data
  • Develop training materials — Onboarding guides for store staff
  • Roll out to broader Zeiss Vision Centre network

Key takeaway: The prototype's clean stage-based architecture makes it naturally "pluggable" — each stage is independent and can be wired to a real backend service without touching the others. Production integration can happen incrementally, one stage at a time, while keeping the app functional throughout.
